# C4 Diagram

## System Context

```mermaid
C4Context
    title CoA - System Context
    Person(user, "Public User", "Views alerts, shelters, routes and events")
    Person(admin, "Administrator", "Manages data in the admin interface")
    System(coa, "CoA", "Web application for emergency-related information")
    System_Ext(mail, "Email Service", "Sends account verification and password reset emails")
    System_Ext(db, "MySQL Database", "Stores users, shelters, routes, events and notifications")

    Rel(user, coa, "Uses")
    Rel(admin, coa, "Manages content through")
    Rel(coa, mail, "Sends emails via")
    Rel(coa, db, "Reads and writes data in")
```

## Container View

```mermaid
C4Container
    title CoA - Container Diagram
    Person(user, "Public User", "Browser")
    Person(admin, "Administrator", "Browser")

    System_Boundary(coa, "CoA") {
        Container(public, "Public App", "PHP", "Handles public pages and account flows")
        Container(api, "API", "PHP", "Serves JSON, feeds, import and export endpoints")
        Container(adminApp, "Admin App", "PHP", "Provides data management screens")
        ContainerDb(db, "MySQL Database", "MySQL", "Stores application data")
    }

    System_Ext(mail, "Email Service", "Sends verification and reset emails")

    Rel(user, public, "Uses")
    Rel(admin, adminApp, "Uses")
    Rel(public, api, "Calls for structured data")
    Rel(adminApp, api, "Calls for CRUD and data management")
    Rel(public, db, "Reads data")
    Rel(adminApp, db, "Reads and writes data")
    Rel(api, db, "Reads and writes data")
    Rel(api, mail, "Sends emails via")
```

## Why the API is separate

The API is kept separate from the user-facing app so page rendering stays simple and the data layer can evolve independently. Public pages only handle presentation, while the API handles JSON, feeds, import/export, and other machine-oriented interactions. That separation also makes admin actions and integrations easier to maintain without mixing them into the user experience.
